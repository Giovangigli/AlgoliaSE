A record is one searchable object in an Algolia index. In your case, one restaurant would usually be one record. Each record contains attributes such as the restaurant name, cuisine type, location, rating, review count, price range, and payment options.

Indexing is the process of sending those records to Algolia so they can be searched quickly. During indexing, Algolia prepares the data for fast retrieval, filtering, ranking, and typo-tolerant search.

Custom Ranking is used to break ties between otherwise relevant results. Good custom ranking metrics are business or quality signals that should influence ordering after textual relevance has been considered.

For restaurants, useful custom ranking metrics could include:

- Average rating
- Number of reviews
- Popularity or booking volume
- Availability
- Price tier, if the business wants to promote a certain segment

Distance should usually be handled through Algolia geo-search rather than Custom Ranking.

Hi Leo,

No, Algolia is designed to be easy to integrate. For a standard implementation, the development effort is usually low, and a lot can be done with low-code configuration once the data structure is clear.

The high-level process is:

1. Prepare your data: Decide what each searchable record should contain. For example, for a restaurant website, each record might include name, cuisine, location, rating, price range, and availability.

2. Send the records to Algolia: Use Algolia’s API or one of the API clients to index your records.

3. Configure relevance: Define searchable attributes, facets, custom ranking, typo tolerance, and geo-search settings.

4. Build the frontend experience: Add a search box, render results, and connect filters such as cuisine, location, or price.

5. Test and iterate: Try real user queries, inspect weak results, and adjust ranking or data structure where needed.

The search experience itself can be implemented quickly. The main work is usually preparing clean data and tuning relevance so the experience matches user intent.
